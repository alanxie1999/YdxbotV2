# 版本与发布规范

本文档用于约束本项目的 tag / release 方式，避免后续混乱。

## 核心原则

1. `Git Tag` 和 `GitHub Release` 使用同一个版本号。
2. 不要为同一次发布维护两套不同版本号。
3. 如果需要区分“预发布”和“正式发布”，用 tag 后缀区分，而不是再造一套 release 版本号。

## 推荐做法

### 正式版本

- Tag：`v1.1.21`
- Release：`v1.1.21`

说明：
- Release 应直接绑定同名 tag。
- 这是最标准、最清晰、最容易追踪的方式。

### 预发布版本

推荐未来统一使用这类形式：

- `v1.1.22-rc.1`
- `v1.1.22-beta.1`

说明：
- `rc` 表示候选发布版。
- `beta` 表示测试版。
- GitHub Release 中应标记为 `prerelease=true`。

### 不推荐做法

- Tag 用 `v1.1.21`，Release 再叫 `release-2026-03`
- Tag 用 `v1.1.21`，Release 再叫 `R20260323`
- 同一版本同时打两个正式 tag

原因：
- 使用者会分不清哪个才是真正版本。
- 自动化脚本、更新说明、回滚定位都会变复杂。

## 建议的发布节奏

### 方案 A：单线发布

适合当前轻量项目，推荐默认使用：

- 日常修复：直接发正式版，如 `v1.1.22`

### 方案 B：双线发布

当后续用户更多、更新更频繁时再启用：

- 预发布线：`v1.1.22-rc.1`
- 正式发布线：`v1.1.22`

## 每次发布最少要做的事

1. 确认工作区干净。
2. 确认测试通过。
3. 更新 `docs/CHANGELOG.md`。
4. 打 tag。
5. 推送 tag。
6. 创建对应 Release。
7. Release 文案写中文，且面向使用者。

## 建议工作流

从 `v1.1.21` 之后，推荐固定按下面流程操作：

1. 开发完成并确认工作区干净。
2. 跑测试。
3. 更新 `docs/CHANGELOG.md`。
4. 提交代码。
5. 打正式 tag，例如 `v1.1.22`。
6. 推送 `main` 和 tag。
7. 创建同名 GitHub Release。
8. Release 文案写“相对上个 tag 的更新内容”。

推荐发布文案结构：

- 第一段：版本号
- 第二段：更新内容
- 第三段：本次版本重点

示例：

```md
v1.1.22

相对 v1.1.21 更新内容
- 修复……
- 优化……

本次版本重点
- 对使用者最重要的变化……
- 需要关注的行为变化……
```

## 历史版本处理

如果以后再发现“tag 已存在，但 release 缺失”：

1. 不要重打 tag。
2. 直接为同名 tag 补建 GitHub Release。
3. Release 标题与 tag 保持一致。
4. 历史补录时，在说明里明确写“历史版本补录”。

## 当前结论

对这个项目来说，最合适的规范是：

- 正式版：`vX.Y.Z`
- 预发布：`vX.Y.Z-rc.N`
- Release 与 Tag 同号，不拆成两套版本体系

这已经足够标准，也不会把轻量项目搞得过重。
